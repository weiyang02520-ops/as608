# 基于 STM32F103 + AS608 + 微信小程序的指纹考勤系统设计

> 本文记录一个完整的嵌入式指纹考勤项目：硬件端基于 STM32F103C8Tx 和 AS608 指纹模块，完成指纹录入、识别、打卡记录、OLED 菜单、EEPROM 持久化；移动端使用微信小程序通过 BLE 连接蓝牙串口模块，实现时间同步和排行榜读取。项目源码已在 GitHub 以 MIT License 开源。

## 一、项目简介

这个项目最初是一个指纹打卡机原型，但后来逐步补上了本机菜单、管理员密码、用户姓名、EEPROM 记录、累计工时统计、蓝牙通信和微信小程序端，最后形成了一套比较完整的“端侧考勤系统”。

项目由两部分组成：

- **STM32 固件端**：负责指纹识别、用户管理、考勤记录、OLED 显示、矩阵键盘输入、RTC 时间维护和 EEPROM 数据保存。
- **微信小程序端**：通过 BLE 连接蓝牙串口模块，向硬件端发送时间同步指令和排行榜请求，并展示累计工时排行。

项目开源地址：

```text
https://github.com/weiyang02520-ops/as608
```

## 二、功能概览

### 1. 硬件端功能

- AS608 指纹模块握手、采图、生成特征、模板合成、模板存储和高速搜索
- OLED 中文界面显示
- 4x4 矩阵键盘输入
- 管理员密码保护
- 指纹录入、删除、清空、改名、查看列表
- 上班 / 下班打卡判断
- AT24C32 EEPROM 保存打卡记录和累计时长
- RTC 时间读取和设置
- 蓝牙串口接收小程序指令

### 2. 小程序端功能

- 扫描并连接蓝牙串口模块
- 自动发现 BLE 服务和特征值
- 向打卡机同步手机当前时间
- 向打卡机请求累计工时排行榜
- 解析 `USER:XX,TOTAL:YYYY` 数据
- 按累计秒数排序并展示排行榜

## 三、硬件清单

## 四、系统架构

整体结构如下：

```text
微信小程序
   |
   | BLE
   v
蓝牙串口模块
   |
   | USART3
   v
STM32F103C8Tx  <---- USART1 ---->  AS608 指纹模块
   |
   +---- I2C ----> AT24C32 EEPROM
   |
   +---- GPIO ---> OLED
   |
   +---- GPIO ---> 4x4 矩阵键盘
```

硬件端可以离线独立运行，小程序端主要用于时间同步和查看排行榜。这样即使没有网络、没有服务器，打卡流程也不会受影响。

## 五、固件端设计

### 1. 启动流程

固件启动后依次完成：

```text
HAL_Init
  -> SystemClock_Config
  -> GPIO / USART / I2C / RTC / TIM 初始化
  -> 蓝牙串口接收初始化
  -> OLED 初始化
  -> AS608 握手
  -> 读取有效模板数
  -> 读取 EEPROM 中的记录数
  -> 初始化管理员密码
  -> 进入主循环
```

主循环中主要处理三类任务：

- 键盘扫描和菜单操作
- 待机状态下扫描指纹
- 蓝牙命令解析

### 2. AS608 指纹流程

录入指纹时，需要两次采集手指：

```text
第一次采图
  -> 生成 CharBuffer1
第二次采图
  -> 生成 CharBuffer2
比对两次特征
  -> 合成模板
  -> 存储到指定 ID
```

打卡时则是：

```text
采图
  -> 生成特征
  -> 高速搜索指纹库
  -> 得到匹配 ID 和分数
  -> 写入打卡记录
```

AS608 的姓名存储使用了模块自带的记事本区域。录入用户时，把姓名写入对应页；打卡成功后，再读取出来显示在 OLED 上。

### 3. 打卡记录与累计时长

项目使用 AT24C32 保存打卡记录。每条记录包含：

- 用户 ID
- 打卡时间
- 打卡类型：上班 / 下班

当同一个用户再次打卡时，程序会根据上一条记录判断当前是上班还是下班。如果本次是下班，就计算和上一次上班之间的时间差，并累加到该用户的总工时中。

当前原型中主要按“当天秒数”计算，因此跨天打卡还需要后续优化。

### 4. 管理员菜单

管理员输入密码后进入后台菜单：

```text
1. Add User      录入指纹
2. Del User      删除指纹
3. Mod Name      修改姓名
4. View List     查看列表
5. Change PWD    修改密码
6. Del All FR    清空指纹
7. Reset Time    重置累计时长
```

默认管理员密码为：

```text
123456
```

实际使用时建议修改。

## 六、蓝牙协议设计

为了方便调试，硬件端和小程序端之间没有使用复杂二进制协议，而是使用简单文本协议。

### 1. 小程序同步时间

小程序发送：

```text
SET_TIME=2026-07-15 09:30:00
```

硬件端解析后写入 RTC。

### 2. 小程序请求排行榜

小程序发送：

```text
GET_RANK
```

硬件端返回：

```text
USER:01,TOTAL:3600
USER:02,TOTAL:5400
```

其中 `TOTAL` 的单位是秒。小程序收到后会换算成“秒 / 分钟 / 小时”的显示格式，并按累计时长排序。

## 七、微信小程序端

小程序源码在仓库中的目录：

```text
wechat-miniprogram/
```

核心逻辑在：

```text
wechat-miniprogram/pages/index/index.js
```

小程序主要做几件事：

1. 打开蓝牙适配器。
2. 扫描名称中包含 `BT24` 或 `DX` 的设备。
3. 建立 BLE 连接。
4. 查找 `FFE0` 服务。
5. 找到可写特征和可通知特征。
6. 监听硬件端返回的数据。
7. 发送同步时间和排行榜请求。
8. 解析排行榜数据并刷新页面。

项目里的 `project.config.json` 使用的是：

```json
"appid": "touristappid"
```

如果要发布自己的小程序，需要换成自己的 AppID。我的测试号 AppID 没有上传到 GitHub。

## 八、项目踩坑记录

### 1. 串口接收不能只靠固定延时

AS608 返回的数据长度不完全一样，如果只用固定延时等待，容易漏包或卡死。项目里使用了 UART 中断和空闲中断来辅助接收。

### 2. EEPROM 地址要提前规划

原型阶段为了快速实现功能，把记录数、密码、打卡记录和累计时长都塞进了 EEPROM。后续如果要长期使用，需要重新设计地址表，避免区域重叠。

### 3. BLE 数据可能分包

小程序收到的数据不一定一次就是完整的一行，所以用了 `receiveBuffer` 拼接数据，等遇到 `\n` 再按行解析。

### 4. AppID 不应该随源码公开

小程序项目开源时，建议把 `project.config.json` 里的 AppID 改成 `touristappid`，并排除 `project.private.config.json` 这种本机私有配置。

## 总结

这个项目不是特别复杂，但覆盖了嵌入式项目里很常见的一条完整链路：传感器采集、本地交互、数据存储、串口通信、移动端展示。对学习 STM32、AS608 指纹模块、EEPROM 存储和微信小程序 BLE 通信来说，还是比较适合作为一个综合练手项目的。
